#  Cognitive Multi-Agent Historical Mapping System Quantum Secured Chat Assistant

## Overview
Gummaa-Atlas is a multi-agent orchestration system designed to provide historical and geospatial context, focusing on Pan-African history, civilisations, and trade routes. Built using Python, it leverages **Google Gemini** for reasoning and content generation and **LanceDB** for a time-travel capable, persistent cognitive memory lakehouse. The system uses a multi-agent architecture (Coordinator, GraphAgent, DiscovAgent, Synthesizer) to break down user queries, correlate historical facts, discover geographical data, and synthesize a coherent, scholarly response.

---

## Complete Implementation Guide

### Prerequisites
- Python 3.9+
- Google Cloud Project with Vertex AI enabled
- Gemini API
- LanceDB

### Setup & Installation
1. **Clone the repository** (or navigate to your working directory).
2. **Install dependencies**: 
   Ensure you have all the required Python packages installed.
   ```bash
   pip install -r requirements.txt
   ```
3. **Environment Variables**:
   Create a `.env` file in the root directory to configure the application:
   ```env
   PROJECT_ID=multi-agent-lab01
   LOCATION=us-central1
   LANCEDB_PATH=./gummaa_workspace.lance
   PORT=8080
   ```
4. **Data Ingestion**:
   Run the ingestion script to pre-populate the LanceDB memory lakehouse with the initial seed data. This gives the system its foundational historical context.
   ```bash
   python ingest.py
   ```
5. **Start the Server**:
   Launch the FastAPI backend server.
   ```bash
   python server.py
   ```
   The backend and frontend will be available at `http://localhost:8080`.

## Architecture & Component Deep Dive

### 1. `server.py` (FastAPI Backend)
The entry point of the application. It acts as the bridge between the frontend map interface and the agent backend. It provides the following RESTful endpoints:
- `GET /` - Serves the static `index.html` frontend.
- `POST /api/chat` - The main endpoint for the conversational interface. It receives the user's message and optional map coordinates, instantiates a session, and triggers the `GummaaAgent` to run the multi-agent pipeline.
- `GET /api/memory` and `DELETE /api/memory` - Used to inspect the state of the cognitive memory lakehouse and prune expired memories.

### 2. `agent.py` (Compute-Bound Orchestrator)
This is the core of the multi-agent workflow. It orchestrates the sub-agents using a safe execution model wrapped in LanceDB time-travel (`.checkout()`). The pipeline consists of four distinct agents:
- **Coordinator**: Parses the initial user query, extracts intent and named entities, and writes a structured task brief for the other agents.
- **GraphAgent**: Cross-correlates the brief with the memory context to produce a narrative knowledge graph of historical relationships, causal chains, and temporal contexts.
- **DiscovAgent**: Extracts places from the brief, resolves them to coordinates using geospatial tools, and builds a GeoJSON overlay to be rendered on the map.
- **Synthesizer**: Merges the outputs from the Coordinator, GraphAgent, and DiscovAgent, along with cognitive memory, to generate a single, scholarly final response for the user.

### 3. `tools.py` (Geospatial Utilities)
Provides specialized geospatial tools used primarily by the `DiscovAgent` to interact with geographical data:
- `geocode()` and `reverse_geocode()`: Interfaces with Nominatim (OpenStreetMap) to translate place names to lat/lng coordinates and vice versa, without needing an API key.
- `build_map_overlay()`: Constructs a GeoJSON FeatureCollection to be rendered seamlessly on the frontend map.
- `bbox_filter()`, `pmtiles_url()`, `haversine_distance()`, and `classify_region()`: Additional geospatial logic to support bounding boxes, vector map tiles, distance calculations, and historical region tagging based on bounding boxes.

### 4. `ingest.py` (Data Ingestion Pipeline)
A standalone script used to seed the LanceDB cognitive memory lakehouse. It contains a curated dataset of Pan-African historical facts, geospatial anchors, and trade route data. When run, it embeds these facts directly into the database as vectors, providing the system with rich contextual knowledge from day one, allowing the agents to pull relevant facts during query resolution.

### 5. Google Gemini (LLM Engine)
Gemini (specifically `gemini-2.5-flash`) powers the reasoning capabilities of all the sub-agents. Accessed via the Vertex AI SDK (`google-genai`), Gemini:
- Reads the prompts generated by the orchestrator.
- Processes the historical context retrieved from LanceDB.
- Generates structured data (like JSON from the DiscovAgent) or scholarly text (like the output from the Synthesizer) depending on the agent's system prompt.

### 6. LanceDB (Cognitive Memory Lakehouse)
LanceDB serves as the persistent, embedded vector database for the system. It enables three critical features:
- **Semantic Search (Recall)**: The agent retrieves relevant historical context before each LLM call based on vector similarity to the user's query.
- **Persistent Memory (Remember)**: The system stores past interactions, allowing it to maintain conversational and analytical context across a session.
- **Time-Travel Checkpoints**: Used by the `safe_execute()` function in `agent.py`, LanceDB allows the system to snapshot its version before a complex operation. If an exception occurs, the system can instantly roll back, ensuring a zero-penalty error recovery mechanism.
